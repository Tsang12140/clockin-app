# AI 助手调优记录

> 本文档记录 `荣源考勤` AI 管家从零搭建到持续迭代的全过程。  
> 供后续 AI 协作者和开发者理解"我们做过什么、为什么这么做"。

---

## 架构概览

```
用户提问
  └─ planWithAI()        ← AI 模型：理解意图，输出结构化 JSON 计划
       └─ executePlan()  ← TypeScript：确定性执行，查数据库，拼回答
            └─ yield AssistantResponse  ← SSE 流式推送到前端
```

核心文件：
- **`lib/ai/attendanceAssistant.ts`** — 全部 AI 逻辑（planner prompt、executor、summarizer）
- **`components/AIAssistant.tsx`** — 前端面板 UI
- **`app/api/ai/assistant/route.ts`** — SSE 路由
- **`lib/ai/factsCache.ts`** — 历史区间数据缓存
- **`app/(main)/actions.ts`** — 服务端 Action，触发缓存失效

## 调优时间线

### 阶段一：两阶段架构设计

**动机**：直接让 AI 回答考勤问题，数字经常算错（幻觉），格式不稳定，无法可靠跳转页面。

**方案**：将 AI 的职责切割为两层：
- **Planner（AI）**：只负责理解意图，输出 JSON 计划（intent + 参数），不做计算
- **Executor（TypeScript）**：根据 JSON 计划查数据库、算数字、拼文本，结果 100% 准确

**效果**：AI 负责"听懂"，代码负责"算对"，两者各司其职，彻底消除数字幻觉。

---

### 阶段二：多意图 + 流式输出

**问题**：用户一句话问两件事（"谁最勤快？谁请假最多？"），AI 只回答一个。

**改动**：
- `planWithAI()` 返回类型从 `AssistantPlan` 改为 `AssistantPlan[]`
- Planner prompt 新增规则：多问题时返回 JSON 数组，每问一个 plan 对象
- `answerAttendanceAssistantStream()` 改为 `AsyncGenerator`，循环 yield 每个 plan 的结果
- 前端改为 SSE 流式接收，每收到一个 `answer` 事件就新增一个气泡

**Prompt 关键规则**：
```
如果用户在一条消息里提了多个独立问题，返回 JSON 数组，
每个问题一个 plan 对象；只有一个问题时返回单个对象。
```

---

### 阶段三：跨月状态查询优化

**问题**："过去三个月谁请过病假" 极慢（3 次 AI 调用：CLARIFY → 生成 SQL → 执行）。

**根因**：`QUERY_RANGE_SUMMARY` 没有 `status` 字段，planner 无法表达"跨月 + 特定状态"的组合意图，被迫走 CLARIFY。

**改动**：
- `AssistantPlan` 新增 `status` 字段
- `summarizeRangeFacts()` 新增 `status` 参数，有 status 时按 sickDays/leaveDays/absentDays 排名，否则按 totalHours

**Prompt 关键规则**：
```
QUERY_RANGE_SUMMARY 不需要 year/month 字段，只需要 startDate/endDate、
metrics，可选 status（病假/请假/旷工等跨月状态查询时设置）。
```

---

### 阶段四：代词上下文推断

**问题**：
- "今年到现在谁最勤快？" → AI 回答了排名
- "这两个人谁入职最久？" → AI 问"请问您指的是哪两个人？"

**根因**：Planner 看不到历史对话，无法把"这两个人"映射到前一条回复里的具体姓名。

**改动**：
- `AssistantPlan` 新增 `employeeNames?: string[]` 字段
- `resolveEmployeeScope()` 优先处理 `employeeNames` 数组
- 新增 `QUERY_EMPLOYEE_PROFILE` 意图处理入职时间/工价比较
- 新增 `summarizeEmployeeProfile()` 函数

**Prompt 关键规则**：
```
如果用户说"这两个人""这几个人""他们""第一和第二"等代词，
必须从 recentConversation 的 assistant 最近回复中提取出具体姓名，
填入 employeeNames 数组（如 ["林一鸣","何二朗"]），不要反问用户是谁。
```

---

### 阶段五：查询结果缓存

**动机**：跨月历史查询（如"去年全年工时"）每次都扫描大量数据库记录，响应慢。

**方案**：文件缓存 `RangeFacts`，键为日期范围，无过期时间，由数据修改触发失效。

**规则**：
- 只缓存 `endDate < today` 的历史区间（当前和未来数据不缓存，因为还在变化）
- `saveAttendance()` 保存考勤后，自动清除涉及日期的缓存文件
- `clearAttendanceDay()` 清空某天数据后，自动清除对应缓存

**缓存文件位置**：`.runtime/facts-cache/range_YYYY-MM-DD_YYYY-MM-DD.json`（已加入 `.gitignore`）

**注意**：缓存省的是数据库查询时间，**不省 AI Token**（planner 每次仍调用）。

---

### 阶段六：自动导航 + 页面上下文推断

**问题一**：`OPEN_EMPLOYEE`、`OPEN_PAYSLIP`、`OPEN_SALARY` 意图只给按钮，用户还要手动点。

**改动**：
- `AssistantResponse` 新增 `autoNavigate?: boolean` 字段
- 三个 OPEN_* 意图在 executor 返回时设置 `autoNavigate: true`
- 前端收到 `autoNavigate: true` 时立即调用 `router.push()`，移动端同时收起面板

**问题二**：在员工详情页（`/employees/8`）说"看上个月工资单"，AI 问"哪位员工？"

**根因**：`pageUrl` 虽然从前端发送，但只存日志，没传入执行逻辑。

**改动**：
- `pageUrl` 从 route.ts 传入 `answerAttendanceAssistantStream()`，再传入 `executePlan()`
- 新增 `employeeIdFromUrl()` 解析 `/employees/[id]` 或 `/salary/[id]` 中的 ID
- `executePlan()` 中：若 plan 未解析出员工，且 pageUrl 含员工 ID，则以 URL 中的员工为兜底

---

### 阶段七：高优先短板修复（v1.87）

**问题一**：在员工详情页问"他这个月工时多少"，planner 仍不知道当前员工是谁（pageUrl 只传到 executor 层）。

**改动**：
- `planWithAI()` 签名加 `pageUrl` 参数
- user content 新增 `currentPageUrl` 字段传给 planner
- prompt 新增规则：URL 含 `/employees/[id]` 或 `/salary/[id]` 时，省略主语的问题直接填 `employeeId`

**问题二**：`isOpenEmployeeIntent` 正则过宽，"员工的工时"被误识别为打开员工资料。

**改动**：正则改为要求"动词（打开/查看）"与"名词（资料/档案/信息）"**同时出现**才触发。

**问题三**：CLARIFY → SQL fallback 不稳定，AI 生成的 SQL 质量参差不齐且有安全风险。

**改动**：去掉整个 SQL fallback 路径，CLARIFY 直接返回固定能力介绍文案。

---

### 阶段八：中优先短板修复（v1.88）

**问题一**：用户问"你能做什么"走到 CLARIFY，回答随机不稳定。

**改动**：新增 `isCapabilityQuestion()` 关键词拦截函数，在 stream 入口最早期拦截，直接返回固定的能力清单，完全不走 AI。

**Prompt 关键规则**：无（纯代码拦截，不消耗 token）

**问题二**：同姓多人时消歧文案生硬（"xxx对应A、B，你想查哪一位？"）。

**改动**：改为带序号的引导式文案——"找到N位：1.A、2.B，请说名字告诉我查哪一位？"，用户可以直接说名字继续对话。

**问题三**：历史对话截断 500 字，长回复在续问时上下文丢失。

**改动**：`text.slice(0, 500)` → `text.slice(0, 800)`。

**问题四**：`QUERY_MISSING` 和 `QUERY_ANOMALY` 触发条件模糊，同一句话可能进不同 intent。

**Prompt 新增规则**：
> QUERY_MISSING 专指"整天没有录入任何人"的日期（全员空白）；QUERY_ANOMALY 指已录入但含异常状态（请假/病假/旷工）或个别人漏录。"有没有漏录""谁没录"→QUERY_MISSING；"有没有异常""谁请假/旷工了"→QUERY_ANOMALY。

---

### 阶段九：低优先短板修复（v1.89）

**问题一**："近期""最近"无具体天数时 AI 自行猜测，结果不稳定。

**Prompt 新增规则**：
> "近期""最近""这段时间" 无具体天数 → 默认近7天；"最近几个月" → 近3个月；"最近半年" → 近6个月；"最近一年" → 近12个月。均使用 QUERY_RANGE_SUMMARY。

**问题二**：所有错误（网络断、429、401、服务异常）统一返回"刚才没有查成功"，用户无法判断原因。

**改动**（前端 `AIAssistant.tsx`）：
- HTTP 429 → "发送太频繁了，稍后再试。"
- HTTP 401 → "登录已过期，请刷新页面重新登录。"
- 网络 TypeError → "网络连接失败，检查一下网络再试。"
- 其他异常 → "查询出错了，稍后再试一下。"

**问题三**：Planner 收不到员工入职日期，无法在计划层直接定位"入职最久的人"。

**改动**：employees 列表加入 `hireDate` 字段，planner 可直接引用。

---

### 阶段十：页面上下文兜底修复（v1.90）

**问题**：在员工详情页问"他上个月工资多少"，AI 仍然反问"请问是哪位员工？"

**根因**：v1.87 只在 prompt 里加了规则要求 planner 推断 employeeId，但 LLM 不可靠地遵守——仍然返回 CLARIFY。pageUrl 传到了 planner，但没有在代码层强制兜底。

**改动**：在 `executePlan` 的 CLARIFY 分支加代码层兜底：
- 若 intent 为 CLARIFY，且 pageUrl 含员工 ID，且问题含考勤关键词（工资/工时/出勤/请假等）
- 直接合成 `QUERY_EMPLOYEE_MONTH` plan，用 URL 员工 ID 执行，跳过反问
- 关键词含"工资/薪资/多少钱" → metrics=['wage']；否则 → metrics=['hours']

**教训**：依赖 LLM 遵守 prompt 规则处理关键路径不可靠。凡是"必须做到"的逻辑，应在 TypeScript 代码层加确定性兜底，prompt 规则只作辅助。

## Planner Prompt 规则全集（当前版本）

以下是 `planWithAI()` 中 system content 数组的全部规则，按逻辑分组：

### 基础框架
- 角色：只输出 JSON，不回答用户，不加 Markdown，不解释
- 允许的 intent：`QUERY_EMPLOYEE_MONTH`, `QUERY_STATUS_RANKING`, `QUERY_MONTH_SUMMARY`, `QUERY_RANGE_SUMMARY`, `QUERY_EMPLOYEE_PROFILE`, `QUERY_MISSING`, `QUERY_ANOMALY`, `OPEN_PAYSLIP`, `OPEN_EMPLOYEE`, `OPEN_SALARY`, `CLARIFY`

### 字段映射规则
- status：`请假=leave`，`病假=sick`，`旷工=absent`，`休息/放假=holiday`，`正常/上班=worked`
- metrics：`hours, wage, count, days, records, ranking, list`
- employeeFilter：`gender, status, positionName, nameContains, minHourlyRate, maxHourlyRate`
- 女员工=`gender:female`；男员工=`gender:male`；在职=`status:active`

### 意图选择规则
- "上班最多/工时最多" → `QUERY_MONTH_SUMMARY + metrics:["hours","ranking"]`，不要理解为 worked 天数
- 异常/漏录/汇总/排行 + 无指定员工 → 默认全员，不反问
- `QUERY_EMPLOYEE_PROFILE`：入职时间/工龄/工价 比较，`metrics: hireDate 或 hourlyRate`，`order: top=最久/最高, bottom=最新/最低`
- `QUERY_RANGE_SUMMARY`：跨月/跨年问题，输出 `startDate/endDate`，可选 `status`

### 时间解析规则
- 无月份 → 当前年月
- "去年" → 前一年同月
- "去年全年" → `startDate=去年01-01, endDate=去年12-31`
- "今年上半年" → `startDate=今年01-01, endDate=今年06-30`
- "今年" → `startDate=今年01-01, endDate=今天`
- "近3个月" → `startDate=3个月前1号, endDate=今天`
- 季度：Q1=01-01至03-31，Q2=04-01至06-30，Q3=07-01至09-30，Q4=10-01至12-31

### 上下文推断规则
- 问题很短（如"那请假呢""他呢"）→ 必须从 `recentConversation` 补全员工、月份、状态
- 代词（"这两个人""他们""第一和第二"）→ 从 assistant 最近回复提取姓名填入 `employeeNames[]`
- 上下文继承（"按刚才的""每个人列出来"）→ 继承 `lastPlan` 的 filter/year/month/status/metrics

### 输出格式规则
- "列出来/每个人/排名/排序" → metrics 必须含 `list` 和 `ranking`
- "谁/哪个 + 最高/最低" → 只想要一个人，metrics 不放 ranking/list
- 能唯一确定员工 → 填 `employeeId`
- 多问题 → 返回 JSON 数组；单问题 → 返回单个对象
- 无法匹配任何 intent → 返回 `CLARIFY + clarifyingQuestion`
